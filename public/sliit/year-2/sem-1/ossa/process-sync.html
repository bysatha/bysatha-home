Race condition

Problem - Concurrent

Atomic: Without interrupt

Non-atomic: With interrupt

This is a non atomic (because of an interrupt): and it may result for data inconsistency, therefore sequential access is the best.

Race condition:

Race condition is a situation where several processes access/manipulate the same data, resource concurrently.

So, the outcome of the execution depends on particular order in which access takes place.

Solution: Process synchronization. - Only one process can manipulate at a time.

Critical section: in this example, counter variable is the critical point.

Critical section Definition:

Critical section is a segment of code which process maybe changing common variables, updating table, writing file, etc.

Now in here, the critical section problem is:

Producer does the ++ operation meanwhile (concurrently) the consumer does the - - operation to the same counter variable.

Solutions for the critical section problem

  1. Mutual exclusion: if p1 process is in the critical section → p2 will not go to the critical section (at the same time)
  1. Progress: if p1 is not in the critical section → it should not block the p2 to enter → to the critical section. (If I ain’t doing something, I should not block others) : Also, Progress means, the requested process should be able to proceed.
  1. Bounded waiting: If a process is waiting (to enter to the critic sec.) → there should be a limit of waiting.

Simple solutions:

  • Disable all interrupts (but as disabling/enabling of interrupts is a privileged operation, it can’t be done every time).
  • Preemptive (no interrupts) kernel: this is not an easy task.

Algorithm 1: Bounded waiting ❌, Progress ❌, Mutual exclusion ✅
